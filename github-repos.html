GitHub Repo Catalog: 15 Curated Open-Source Tools

High-confidence, production-maturity recommendations to close the gaps identified in the plan evaluation. Every repo here is widely used, actively maintained, and integrates cleanly with the existing Python/BigQuery stack. No experimental repos.

Selection criteria: Each repo had to meet four bars: (1) actively maintained in 2025-2026, (2) production-tested at scale (used by Microsoft, Uber, Google, or large open-source community), (3) integrates with Pandas/scikit-learn/BigQuery, (4) clear documentation and examples. Experimental, single-author, or research-only repos were excluded.

MLOps Foundation (Repos 1-4)

1. mlflow/mlflow Production High Impact

github.com/mlflow/mlflow · 19K+ stars · Maintained by Databricks · Used at scale by Microsoft, Booking.com, Toyota

What it does: Experiment tracking, model registry, model serving. Logs every model training run with hyperparameters, metrics, artifacts, and source code commit hash.

Why you need it: The plan trains 10+ models across multiple validation windows. Without MLflow, you cannot answer "which run produced this score?" or "can we reproduce yesterday's result?" Data science review will reject any work that lacks experiment tracking.

Integration: Two-line setup. Wrap every fit() call with with mlflow.start_run():. Logs to local SQLite by default; can be switched to BigQuery-backed for team use.

Risk it mitigates: Reproducibility. Audit trail. Model versioning.

2. great-expectations/great_expectations Production High Impact

github.com/great-expectations/great_expectations · 10K+ stars · Maintained by GX Cloud · Used by Heineken, ING, Vimeo

What it does: Data validation framework. Define expectations ("current_mrr is non-null and > 0", "row count is within 5% of yesterday") as code. Run as part of the pipeline. Fail fast on violations.

Why you need it: The plan reads from 30+ BigQuery tables. If a column changes type, a feed breaks, or a value distribution shifts, the pipeline silently continues with broken assumptions. GX catches this at the data layer before models train on bad input.

Integration: Define a "checkpoint" per data source. Run as the first step of each src/*.py script. Failures halt the pipeline and produce a diagnostic report.

Risk it mitigates: Silent data corruption. Models trained on bad data. Downstream errors that look like model bugs.

3. feast-dev/feast Production Medium Impact

github.com/feast-dev/feast · 5.7K+ stars · Originally Gojek/Google · Used by Robinhood, Twitter, Discord

What it does: Open-source feature store. Defines features as code, versions them, serves them consistently between training and inference. BigQuery offline store + Redis online store is the common pattern.

Why you need it: Features in Phase 3 are computed in BigQuery but not versioned. If "30-day Builder sessions" changes definition mid-project, you can't compare today's scores to yesterday's. Feast (or even a YAML + hash approach) prevents this.

Integration: Higher overhead than the others. If team is small, start with a hash-of-SQL approach in feature_definitions.yaml. Adopt Feast when scaling to multiple model consumers.

Risk it mitigates: Feature drift between training and scoring. Inability to reproduce historical features.

4. evidentlyai/evidently Production Medium Impact

github.com/evidentlyai/evidently · 5K+ stars · Maintained by Evidently AI · Used by Hyatt, Toyota Research

What it does: Model and data drift monitoring. Compares production data distributions to training distributions; flags feature drift, prediction drift, and target drift.

Why you need it: Phase 17 writes scores to BigQuery, but there's no plan for what happens when models silently degrade. Evidently catches: feature distribution shifts (new customer mix), prediction calibration drift, target drift (engagement metrics changing).

Integration: Run as a scheduled BigQuery job weekly. Outputs a drift report HTML. Alert when drift score > threshold.

Risk it mitigates: Models silently going stale post-deployment. Drift not detected until the business notices.

Survival Analysis & CLV (Repos 5-6)

5. CamDavidsonPilon/lifelines Production High Impact

github.com/CamDavidsonPilon/lifelines · 2.4K+ stars · Maintained by Cam Davidson-Pilon · Used in financial services, healthcare, churn analysis

What it does: Survival analysis library. Kaplan-Meier curves, Cox proportional hazards, Aalen additive models, Weibull AFT. The standard Python library for time-to-event analysis.

Why you need it: This closes Fix #1 from the evaluation. Replace 12 × MRR with actual retention curves by segment. KaplanMeierFitter gives you "expected retention months over the next 12 months" per segment.

Integration:

from lifelines import KaplanMeierFitter
kmf = KaplanMeierFitter()
kmf.fit(durations, event_observed)
expected_months = kmf.survival_function_.iloc[:12].sum().item()
clv = mrr * expected_months

Risk it mitigates: Overstating retention value by 30-50%. The single largest source of revenue inflation in the original plan.

6. sebp/scikit-survival Stable Medium Impact

github.com/sebp/scikit-survival · 1.2K+ stars · Used in healthcare and reliability engineering

What it does: sklearn-compatible survival models. Cox PH, Random Survival Forest, Gradient Boosted Survival Analysis. Provides feature-conditional survival predictions, not just group-level KM curves.

Why you need it: KaplanMeier (from lifelines) gives you average retention by segment. scikit-survival gives you individual customer retention conditioned on their features. Use when segments are too coarse.

Integration: Drop-in sklearn API. fit(X, y) where y is structured (event_indicator, time_to_event).

Risk it mitigates: Coarse segment-level retention when individual-level signal is available.

Propensity & Calibration (Repos 7-8)

7. optuna/optuna Production Medium Impact

github.com/optuna/optuna · 11K+ stars · Maintained by Preferred Networks · Used at PyTorch ecosystem scale

What it does: Bayesian hyperparameter optimization with TPE (Tree-structured Parzen Estimator). Smarter than grid search; converges faster than random search.

Why you need it: Phase 7 uses LightGBM/XGBoost. These have 10+ tunable hyperparameters. Without HPO, you're either accepting defaults (underfit) or doing manual grid search (slow, arbitrary).

Integration: Define an objective(trial) function that returns AUC at top decile (not overall AUC — for targeting, top-decile lift matters more). Run 100 trials. Take best params.

Risk it mitigates: Suboptimal model performance. Defendable answer to "did we tune the model properly?"

8. interpretml/interpret Stable Medium Impact

github.com/interpretml/interpret · 6.2K+ stars · Maintained by Microsoft Research · Used in healthcare, credit, hiring

What it does: Explainable Boosting Machine (EBM) — a glass-box gradient boosting model that's inherently interpretable. Comparable accuracy to LightGBM with built-in interpretability (no need for SHAP post-hoc).

Why you need it: SHAP-on-LightGBM is interpretable, but only after the fact. EBM produces directly interpretable feature contributions. For executive review, "this customer scored high because their MRR is in the 90th percentile and their Builder sessions dropped 40% in the last 30 days" is more defensible than a SHAP plot.

Integration: Train EBM alongside LightGBM. If AUC gap is <0.02, use EBM as the primary model for transparency. Otherwise use LightGBM with SHAP.

Risk it mitigates: Black-box model objections in executive review. Regulatory exposure if model is used for tiered customer treatment.

Uplift & Causal Inference (Repos 9-10)

9. maks-sh/scikit-uplift Stable Medium Impact

github.com/maks-sh/scikit-uplift · 800+ stars · Used in marketing analytics at multiple Fortune 500 firms

What it does: sklearn-compatible uplift modeling. Implements T-Learner, S-Learner, X-Learner, R-Learner. Has built-in uplift curve plotting and Qini coefficient.

Why you need it: CausalML and EconML (already in the plan) are powerful but have a steeper learning curve. scikit-uplift is simpler for the common case (estimate CATE from observational treatment data) and produces cleaner visualizations.

Integration: Drop-in sklearn API. Use alongside causalml for cross-validation of methods.

Risk it mitigates: Slower iteration on uplift modeling experiments. Inconsistent visualizations across model types.

10. DoubleML/doubleml-for-py Stable High Impact

github.com/DoubleML/doubleml-for-py · 600+ stars · Academic + industry use · Implements Chernozhukov et al. 2018

What it does: Double Machine Learning for honest causal inference. Cross-fitting, orthogonalization, valid confidence intervals on CATE estimates — even when using flexible ML models.

Why you need it: Phase 8 says "estimate confidence intervals where possible." With T-Learner or X-Learner, the confidence intervals are biased when ML models are used (overfitting in the nuisance functions). DoubleML produces valid CIs.

Integration: Wrap a regression model and a propensity model, both sklearn-compatible. DoubleML handles the cross-fitting and orthogonalization.

Risk it mitigates: Overconfident uplift estimates. Phase 8 confidence intervals that don't actually cover the true effect.

Explainability & Fairness (Repos 11-12)

11. interpretml/DiCE Stable Medium Impact

github.com/interpretml/DiCE · 1.5K+ stars · Maintained by Microsoft Research

What it does: Counterfactual explanations. For any customer, finds "what would need to change for this customer to convert?" Generates multiple diverse counterfactuals, optionally with feasibility constraints (e.g., "MRR can change, but country can't").

Why you need it: SHAP tells you which features mattered. DiCE tells you what specific actions would change the outcome. For GTM activation, counterfactuals are more directly usable than feature importance.

Integration: Wrap a trained sklearn-compatible model. Generate 3 counterfactuals per top-100 customer. Translate to plain-English reason codes.

Risk it mitigates: Recommendations that aren't directly actionable. GTM teams getting SHAP plots they can't operationalize.

12. fairlearn/fairlearn Production High Impact

github.com/fairlearn/fairlearn · 2K+ stars · Maintained by Microsoft / community · Used in regulated industries

What it does: Fairness assessment and mitigation. Computes disparate impact ratios, equalized odds, demographic parity. Provides mitigation algorithms (reweighing, threshold optimization).

Why you need it: Plan tier, country, industry, and tenure are correlated with targeting outcomes. Without explicit fairness analysis, you may inadvertently target in ways that create regulatory or PR risk (e.g., 80% of top decile is one country).

Integration: Run as a Phase 15 reporting step. Output disparate impact ratios across protected/sensitive attributes. Flag any >1.25 or <0.8.

Risk it mitigates: Regulatory exposure. PR risk. Disparate impact lawsuits in geographies where applicable.

Optimization & Sequencing (Repo 13)

13. cvxpy/cvxpy Production Medium Impact

github.com/cvxpy/cvxpy · 5.7K+ stars · Stanford / community · Used in finance, energy, robotics

What it does: Convex optimization modeling. More flexible than PuLP (linear only) — supports quadratic penalties, conic constraints, mixed integer.

Why you need it: The plan's Phase 11 uses PuLP for sequencing. PuLP is fine for pure linear assignment. If you want to add quadratic penalties (e.g., penalize over-allocating to one motion), or have non-linear constraints, cvxpy is more expressive.

Integration: Drop-in for PuLP. Same problem definition style with slightly different syntax. Solver-agnostic (uses ECOS, SCS, GLPK, MOSEK).

Risk it mitigates: Inability to express realistic operational constraints (capacity ceilings, motion balance, fairness).

Reference Architectures (Repos 14-15)

14. GoogleCloudPlatform/marketing-analytics-jumpstart Stable Medium Impact

github.com/GoogleCloudPlatform/marketing-analytics-jumpstart · Google Cloud reference architecture

What it does: End-to-end marketing analytics pipeline on GCP: BigQuery + Vertex AI + Looker. Includes propensity models, customer segmentation, audience activation, and reporting layers.

Why you need it: Closest reference architecture to your actual stack (BigQuery-first). Use it to validate your data flow, scoring pipeline structure, and BigQuery writeback patterns against a Google-blessed reference.

Integration: Reference only. Don't fork. Pull design patterns: how they structure BigQuery tables for feature serving, how they handle Vertex AI training, how they wire the activation layer.

Risk it mitigates: Inventing patterns Google has already published. Audit-trail gaps that are solved problems elsewhere.

15. microsoft/MLOpsPython Stable Medium Impact

github.com/microsoft/MLOpsPython · Microsoft reference for MLOps patterns

What it does: Reference repo for MLOps in Python. CI/CD for ML, model registry patterns, deployment automation, drift monitoring integration.

Why you need it: Reference for how to structure the project's CI/CD around ML artifacts. Includes patterns for: model versioning, test pyramids for ML code, deployment gates.

Integration: Reference only. Pull the CI workflow patterns and test structures. Don't blindly fork.

Risk it mitigates: Inventing MLOps patterns from scratch. CI/CD configurations that miss important ML-specific checks.

What's Already in the Plan (No Changes Needed)

The original requirements.txt already includes several excellent tools. Don't add to them, just use them properly:

  • causalml (Uber) — primary uplift modeling library. Augmented by scikit-uplift (#9) for simpler API and DoubleML (#10) for valid CIs.
  • econml (Microsoft) — alternative uplift modeling. Use causalml as primary; econml for cross-validation of estimates.
  • shap — feature importance and per-prediction attribution. Pair with DiCE (#11) for actionable explanations.
  • lightgbm, xgboost — gradient boosting. Pair with Optuna (#7) for HPO and interpretML EBM (#8) for transparency comparison.
  • hdbscan, umap-learn — density clustering and visualization. Already good choices.
  • pulp — linear optimization for sequencing. cvxpy (#13) is more flexible for non-linear cases.
  • yellowbrick — ML visualization. Use for calibration plots and cluster validity.
How to use this catalog: Don't try to add all 15 repos at once. Start with the 4 MLOps Foundation repos (#1-4) and the survival analysis pair (#5-6). Those close the highest-impact gaps. Add the rest as the project matures and specific needs emerge.
