12-Week Reliability Path: From Plan to Defensible $21M

Week-by-week execution sequence with explicit data gates between phases. The principle: if a gate fails, stop and address the issue rather than carrying broken assumptions forward. This is what turns a smart analytical plan into a high-probability $21M business case.

How to read this: Each week has objectives, deliverables, and a gate at the end. The gate is a hard checkpoint — you do not proceed to the next week until the gate passes. If the gate fails, you fix the issue or escalate. No silent continuation.

Week 1-2 (Foundation)

MLOps Foundation + Schema Discovery

Objectives:

  • Initialize MLflow tracking server (local file-based store is fine for now)
  • Set up Great Expectations and define data contracts for the top 20 BigQuery tables identified in schema discovery
  • Create config/run_metadata.yaml capturing git commit, BigQuery snapshot date, model versions, seeds
  • Run schema discovery, produce outputs/data_dictionary.md with column-level quality scores
  • Document the 41 data domains from the plan (customers, billing, Builder behavior, etc.) with table mappings

Deliverables:

  • great_expectations/ directory with checkpoints for at least 10 critical tables
  • outputs/data_dictionary.md with quality scores per column (null rate, cardinality, freshness, drift)
  • MLflow experiment builder_21m_targeting created and accessible
  • One trial MLflow run logged end-to-end as a smoke test
Gate 1 (End of Week 2):
  • Data dictionary covers ≥90% of fields needed for Phase 3 features
  • GE suite passes for at least the 10 most critical tables
  • At least one feature can be traced end-to-end from raw event to scoring (proof of feature lineage)
  • MLflow logs a complete trial run with hyperparameters, metrics, and artifacts
If this gate fails: Spend an additional week on schema discovery. Do not proceed to feature engineering on shaky data foundations.
Week 3-4 (Features)

Customer Feature Table with Quality Gates

Objectives:

  • Build the Phase 3 customer feature table in BigQuery (8 feature groups: profile, intent, activation, friction, marketing maturity, value/risk, trend, eligibility)
  • Hash every feature SQL definition; store as feature_definitions.yaml with version + date
  • Implement Phase 4 target labels with explicit leakage checks (feature_date ≤ scoring_date AND label_date > scoring_date)
  • Run a cohort sanity check: compute key features for a known cohort 90 days ago and compare to dashboard reality (must be within 5%)
  • Backtest each feature: does the feature value 90 days ago predict the outcome at 60 days ago?

Deliverables:

  • BigQuery customer feature table with 1 row per customer-scoring-date
  • feature_definitions.yaml with SQL hashes, descriptions, owner, modeling readiness flag
  • Leakage check report: 0 violations
  • Cohort sanity check report: all features within 5% of dashboard truth
Gate 2 (End of Week 4):
  • Feature table contains ≥1M customers with <5% missingness on critical features
  • Zero leakage violations across all (feature, label) pairs
  • Cohort sanity check passes (within 5% of dashboard truth) for all critical features
  • GE suite passing on the feature table
If this gate fails: Investigate failing feature individually. Common issues: window definition off by one day, event filter too narrow, partition pruning wrong.
Week 5-6 (Descriptive + Propensity)

Descriptive Segmentation + Propensity Models

Objectives:

  • Run Phase 5 descriptive segmentation with minimum segment size = 100, significance test on key metrics
  • Train Phase 7 propensity models: 10 models (activation, completion, send, conversion, upgrade, churn, etc.)
  • Use Optuna for HPO (100 trials per model, optimize for top-decile lift, not overall AUC)
  • Train interpretML EBM alongside LightGBM; if AUC gap <0.02, prefer EBM for transparency
  • Calibrate every model with isotonic regression; produce calibration plots
  • Run baseline rules to beat: highest MRR first, highest Builder sessions first, plan-tier-only, random

Deliverables:

  • outputs/baseline_segment_metrics.csv with significance-tested metrics per segment
  • 10 trained propensity models registered in MLflow with hyperparameters and metrics
  • Calibration plots for each model (predicted vs actual by decile)
  • Lift-over-baseline report: model lift vs each of the 4 baselines, by decile
Gate 3 (End of Week 6):
  • Top-decile lift > 2x baseline for at least 7 of 10 propensity models
  • Brier score < 0.25 (well-calibrated) for at least 7 of 10 models
  • No model has AUC < 0.65 (random would be 0.50; below 0.65 means there's no signal)
  • Models beat all 4 baseline rules on top-decile lift
If this gate fails: Models that don't beat baselines may indicate insufficient features, label noise, or fundamental predictability limits. Investigate before adding more model complexity.
Week 7-8 (Uplift)

Uplift Modeling + Treatment Effect Estimation

Objectives:

  • Identify all historical treatment data: feature flag exposure, beta participation, lifecycle campaign exposure, A/B test rollouts
  • For each treatment: check sample size (n per arm) and covariate balance (SMD < 0.1)
  • Apply the sample-size rules from the Improvements page: estimate CATE only when n ≥ 1000 per arm; otherwise label as Directional or Assumption Based
  • Use causalml (T-Learner, X-Learner, Causal Forest) as primary; DoubleML for honest confidence intervals; scikit-uplift for visualization
  • Where no treatment data exists, document low/base/high uplift scenarios with clear assumptions
  • Every customer gets one of 4 uplift quality labels: Validated, Directional, Assumption, Insufficient

Deliverables:

  • Uplift estimate per customer per eligible initiative with quality label
  • Covariate balance report for each treatment
  • Sensitivity bounds for CATE estimates (e.g., E-value)
  • Documented assumptions for Assumption-Based estimates (scenarios with rationale)
Gate 4 (End of Week 8):
  • Every customer-initiative pair has an uplift estimate with quality label
  • Validated uplift estimates have valid CIs (from DoubleML or bootstrap)
  • Assumption-Based estimates have written rationale + low/base/high scenarios
  • SUTVA assumption is explicitly checked (treatment doesn't contaminate control)
If this gate fails: The most common failure here is "we thought we had treatment data but it was contaminated." Better to label more estimates as Assumption-Based and be honest than to overclaim causal effects.
Week 9 (CLV + Confidence)

Survival-Based CLV + Confidence Discount Math

Objectives:

  • Compute Kaplan-Meier survival curves by segment using lifelines
  • For each segment, integrate area under survival curve over 12 months to get expected retention months
  • Replace Phase 9 12 × MRR formula with expected_retention_months × MRR
  • Document the confidence discount formula explicitly:
    confidence_discount = (sample_size_score × treatment_evidence_score × calibration_score)^(1/3)
  • Implement each score component as a documented function
  • Update the EV formula to include dependency_score and trust_risk_score from the initiative config

Deliverables:

  • Survival curves chart per segment (paid/free/trial × ecommerce/non × tenure band)
  • Updated revenue value function using survival-based retention
  • Confidence discount formula documented in technical_appendix.md with worked examples
  • EV formula updated to incorporate dependency and trust risk multipliers
Gate 5 (End of Week 9):
  • Survival curves produced for ≥10 customer segments
  • Expected retention months computed per segment (must be reasonable: paid steady-state 8-11, trial 4-7, free 5-9)
  • Confidence discount produces a number in [0,1] for every customer, traceable to inputs
  • EV formula reviewed and signed off by a peer with statistics/ML background
If this gate fails: If retention curves look implausible, check for survivorship bias (computed only on surviving customers) or cohort definition errors.
Week 10 (EV + Monte Carlo + Sequencing)

Expected Value Scoring + Monte Carlo + Sequencing

Objectives:

  • Compute EV per customer per eligible initiative using the full formula
  • Run Monte Carlo simulation (10,000 iterations) on the $21M projection; sample from distributions for P(success), uplift, revenue
  • Report P10/P50/P90 of total aggregate EV
  • Run PuLP/cvxpy optimization with the customer-overlap constraint: each customer assigned to at most one initiative
  • Apply quantitative Wave 1-4 thresholds (P(success), confidence, EV cutoffs)
  • Produce outputs/customer_scores.csv, outputs/segment_scores.csv, outputs/initiative_sequence.csv

Deliverables:

  • $21M projection with P10/P50/P90 confidence band
  • Monte Carlo histogram chart
  • Wave 1-4 assignments with no customer appearing in more than one wave
  • Sequence optimization report: total customers, total EV, capacity utilization per motion
Milestone (End of Week 10): The $21M number now has a confidence interval. The recommendation is no longer "we project $21M" but "we project a median of $X with an 80% confidence interval of [Y, Z]."
Gate 6 (End of Week 10):
  • Monte Carlo produces a sensible P10/P50/P90 (P50 should be within 30% of the deterministic point estimate)
  • No customer appears in more than one wave assignment
  • Wave 1 has at least 1000 customers and a defendable EV-per-customer
  • Cumulative EV across all waves ≥ $21M at the P50 level (if not, document the gap)
If this gate fails: If P50 is far below $21M, the gap analysis becomes the headline finding. Don't massage numbers to hit $21M. Report what the data supports.
Week 11 (Validation)

Backtesting + Fairness Audit + Drift Baselines

Objectives:

  • Run Phase 14 backtests: lift by decile, revenue by decile, cluster stability across time windows
  • Cluster stability check: Adjusted Rand Index across monthly samples; ARI ≥ 0.6 to consider clusters stable
  • Audit top 100 customer recommendations: do reason codes make business sense? Are the same customers showing up repeatedly?
  • Fairness audit with fairlearn: disparate impact ratios across plan tier, country, industry, tenure
  • Set up Evidently drift monitoring baselines for feature distributions and predictions
  • Sensitivity analysis: how does $21M change if confidence discount is 20% lower? If uplift is 20% lower? If revenue values are 20% lower?

Deliverables:

  • Backtest report: lift, revenue, calibration, cluster stability
  • Fairness audit: disparate impact ratios, flagged segments
  • Sensitivity analysis showing $21M elasticity to key assumptions
  • Evidently drift baselines documented and ready for production monitoring
Gate 7 (End of Week 11):
  • Top-decile customers in backtest realized > 2x baseline revenue
  • No disparate impact ratio > 1.25 or < 0.8 on any protected attribute (or, if flagged, documented mitigation plan)
  • Sensitivity analysis shows $21M is reasonably robust to ±20% changes in key assumptions
  • Cluster ARI ≥ 0.6 across time windows (or unstable clusters explicitly excluded from final recommendation)
If this gate fails: A weak backtest is the most important failure to address. If top-decile customers in backtest didn't outperform, the entire model needs to be re-examined. Don't go to executive readout with a model that fails its own validation.
Week 12 (Readout)

Executive Readout + BigQuery Writeback

Objectives:

  • Produce outputs/executive_summary.md with: recommendation, top segments, sequence, P10/P50/P90 revenue, confidence level, key assumptions, risks, mitigations, data gaps, next decisions
  • Produce outputs/technical_appendix.md with: data sources, feature definitions, model choices, validation results, formulas, methodology
  • Run Phase 17 BigQuery writeback with model_version, feature_set_version, scoring_pipeline_version, data_snapshot_date columns
  • Generate Phase 16 visual outputs: cluster map, revenue waterfall (Monte Carlo), lift charts, calibration, SHAP, survival curves, fairness dashboard
  • Conduct dry-run executive readout with a stand-in audience (data science peer, product peer)
  • Iterate on the readout based on feedback before the real presentation

Deliverables:

  • Executive summary (1-page) + supporting deck
  • Technical appendix (full methodology)
  • BigQuery writeback table populated and queryable
  • All Phase 16 visuals saved to outputs/charts/
  • Phase 19 acceptance criteria checklist all green
Final Milestone (End of Week 12): Defensible $21M business case. Executive readout delivered. BigQuery writeback live. The work is reproducible, validated, audit-ready, and has a clear path to operationalization.

What Could Still Go Wrong

Even with all gates passed, here are the failure modes to watch for:

  • Concept drift between training and FY27 execution. Klaviyo shipped Universal Content in Spring 2026. Customer Builder behavior may shift. The model trained on pre-shift data may not predict post-shift behavior accurately. Plan a retrain cadence (quarterly minimum).
  • Roadmap slippage. The plan's $21M assumes initiatives ship on schedule. If Q1 initiatives slip to Q3, the model's wave assignments are stale. Build in a re-sequencing checkpoint after each roadmap milestone.
  • Wave 1 saturation. The model may identify 5,000 customers in Wave 1, but GTM/CSM capacity may be 500. Without capacity-aware sequencing, the top 4,500 don't get reached. Validate motion capacity early.
  • Counterfactual creep. Once customers are targeted, they're no longer counterfactual to themselves. Future backtests need a held-out control group, not just historical comparison.
  • Goodhart's Law on activation. If "30-day campaign send" becomes the optimization target, the model may target customers whose first send was inevitable. Watch the gap between propensity and uplift; if it narrows, the model is targeting people who would have acted anyway.
