Phase 1: Schema Discovery

Add column-level quality scoring to the data dictionary Change

The plan's data dictionary captures field metadata. Add a quality score column:

  • Null rate (% of rows where field is null in the last 30 days)
  • Cardinality (unique values; flag if <2 or near row-count for categorical use)
  • Freshness lag (days between max event date and today)
  • Drift score (KS test vs. the same field 90 days ago)

Phase 3: Customer Feature Table

Add feature lineage and statistical guardrails Change

  • Window leak check: For every feature with a time window (e.g., "30-day Builder sessions"), validate that no feature uses post-scoring-date data.
  • Cohort cohort sanity check: Compute the feature for a known cohort 90 days ago and compare to dashboard reality. If off by >5%, stop.
  • Feature SQL fingerprinting: Hash each feature's SQL definition; store in feature_definitions.yaml. Changing the SQL invalidates historical scores.
  • Trend features: Use ratio + delta + Z-score, not just delta. A 50% drop on 100 sessions is different from a 50% drop on 2 sessions.

Phase 5: Descriptive Segmentation

Add minimum segment size + statistical significance gates Change

For every descriptive cut (paid × tier × tenure), require:

  • Minimum segment size: n ≥ 100 customers (configurable). Smaller segments get rolled into "Other" to prevent noise.
  • Significance test on key metrics: confidence interval around activation rate, churn rate, MRR. Don't report segment differences as material if CIs overlap.

Phase 6: Advanced Clustering

Add decision-tree-on-cluster-labels for explainable rules Change

The plan names clusters manually ("High value frustrated power users"). Replace with a programmatic step:

  1. Run clustering as planned (HDBSCAN, GMM, K-means).
  2. Fit a shallow decision tree (max_depth=3-4) with cluster label as target, features as predictors.
  3. Extract leaf rules ("if Builder_sessions_30d > 5 AND MRR_band == 'high' AND friction_score > 0.7 then Cluster_3").
  4. Name clusters using the dominant leaf rule, not by eyeballing centroids.

This produces if-then rules that are reproducible, testable, and translatable to product/GTM teams.

Phase 7: Propensity Modeling

Add Optuna for HPO and interpretML as glass-box alternative Change

  • Optuna study for each propensity model: 100 trials, time-based cross-validation, optimize for AUC at top decile (not overall AUC).
  • interpretML Explainable Boosting Machine as a second model that is inherently interpretable. Compare AUC vs LightGBM. If close (< 0.02 AUC gap), use EBM for transparency.
  • Calibration step: use isotonic regression to calibrate output probabilities. Uncalibrated XGBoost probabilities are not directly comparable across customers.
  • Calibration plot mandatory output: predicted vs actual by decile, with Brier score.

Phase 9: Revenue Value Modeling

Replace 12×MRR with survival-based CLV Change

This is Fix #1 from the evaluation. The hardcoded 12 × MRR overstates retention value because real customers churn at varying rates.

New approach:

  1. Use lifelines.KaplanMeierFitter to compute survival curves by segment (plan tier × tenure band × ecommerce flag).
  2. For each segment, integrate the area under the survival curve over 12 months to get expected retention months.
  3. Expected retention value = MRR × expected retention months.

Example formula:

from lifelines import KaplanMeierFitter

kmf = KaplanMeierFitter()
kmf.fit(durations=df_segment['tenure_months'], event_observed=df_segment['churned'])
expected_retention_months = kmf.survival_function_.iloc[:12].sum().item()
expected_value = current_mrr * expected_retention_months

For paid customers with 50% retention at month 12, this produces ~8 × MRR, not 12 × MRR. For high-tenure customers with 80% retention, it produces ~10-11 × MRR. For trial/new customers with 30% retention, it produces ~5 × MRR.

Use scikit-survival when you want feature-conditional survival models (e.g., RandomSurvivalForest) for individual customer expected retention.

Phase 10: Expected Value Scoring

Define confidence discount math explicitly Change

This is Fix #3 from the evaluation. The plan uses confidence_discount in the EV formula but never defines it. Use this:

def confidence_discount(customer):
    sample_score = min(1.0, log(n_similar_customers + 1) / log(1000 + 1))
    treatment_score = {
        "validated": 1.0,
        "directional": 0.7,
        "assumption": 0.4,
        "insufficient": 0.2
    }[customer.uplift_quality_label]
    calibration_score = 1.0 - abs(predicted_prob - actual_rate_in_decile)
    return (sample_score * treatment_score * calibration_score) ** (1/3)

The geometric mean prevents any single weak component from dominating, and gives a number in [0,1] that's interpretable as "what fraction of the raw EV should we believe."

Phase 11: Sequence Optimization

Add customer overlap constraint to prevent double-counting Change

This is Fix #5 from the evaluation. The plan optimizes sequencing but doesn't address the case where customer X is assigned to multiple initiatives.

PuLP formulation:

from pulp import LpProblem, LpMaximize, LpVariable, lpSum, LpBinary

prob = LpProblem("BuilderTargeting", LpMaximize)

# x[c,i] = 1 if customer c is assigned to initiative i
x = {(c, i): LpVariable(f"x_{c}_{i}", cat=LpBinary)
     for c in customers for i in eligible_initiatives[c]}

# Objective: maximize total EV
prob += lpSum(EV[c, i] * x[c, i] for (c, i) in x)

# Constraint: each customer assigned to at most one initiative
for c in customers:
    prob += lpSum(x[c, i] for i in eligible_initiatives[c]) <= 1

# Constraint: per-initiative capacity (Wave 1 = first N customers)
for i in initiatives:
    prob += lpSum(x[c, i] for c in customers_for(i)) <= capacity[i]

# Constraint: roadmap quarter availability
for c, i in x:
    if i.quarter == "Q1" and current_quarter != "Q1":
        prob += x[c, i] == 0

prob.solve()

Use cvxpy if you need non-linear constraints (e.g., quadratic penalties for over-allocating to one motion).

Replace qualitative wave thresholds with numeric cutoffs Change

Wave 1 = "P(success) ≥ 0.6 AND confidence_discount ≥ 0.7 AND EV_base ≥ $X". Wave 2 = lower thresholds. Wave 3 = experiment required. Wave 4 = strategic / future.

Phase 12: $21M Waterfall

Add Monte Carlo simulation for confidence intervals Add

This is Fix #4 from the evaluation. Replace the point estimate with a P10/P50/P90 distribution.

import numpy as np

n_simulations = 10000
results = []

for _ in range(n_simulations):
    total = 0
    for customer, init in assignments:
        # Sample from distributions over each EV component
        p_success_sample = np.random.beta(p_success_alpha, p_success_beta)
        uplift_sample = np.random.normal(uplift_mean, uplift_se)
        revenue_sample = np.random.lognormal(revenue_log_mean, revenue_log_sd)
        ev_sample = (eligibility * reachability *
                     p_success_sample * uplift_sample *
                     revenue_sample * confidence_discount)
        total += ev_sample
    results.append(total)

p10, p50, p90 = np.percentile(results, [10, 50, 90])
print(f"$21M projection: P10={p10/1e6:.1f}M, P50={p50/1e6:.1f}M, P90={p90/1e6:.1f}M")

Report all three numbers. "We project a median of $19M with an 80% confidence interval of $14M-$24M" is more honest and more decision-useful than "We project $21M."

Translate each counterfactual to a reason code: "If this customer had 3 more Builder sessions in the next 30 days, P(conversion) would jump from 0.18 to 0.62." That's a CSM playbook, not just a model explanation.

Phase 14: Backtesting

Add lift over baseline rules + CUPED variance reduction Change

  • Baseline rules to beat: highest MRR first, highest Builder sessions first, random selection, plan-tier-only rule. Your model must beat each of these on decile lift, not just AUC.
  • CUPED (Controlled-experiment Using Pre-Experiment Data): use pre-period covariates to reduce variance in backtest comparisons. Standard in A/B test analysis at Microsoft/Netflix.
  • Cluster stability: rerun clustering on monthly samples; measure Adjusted Rand Index across runs. ARI < 0.6 means clusters are not stable enough to use as named segments.
